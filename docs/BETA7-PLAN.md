# Beta 7 — planning draft

**Status: Beta 7 done ✅** (2026-07-18). Nooks exist — a person's own shelf is now organized and presented through them, not a single flat grid.

## Goal

Flagged early (2026-07-17) as potentially **the apex of the whole product**: the shelf is the one surface entirely about a person's own taste and judgment, and it didn't reflect that at all — just a flat, chronological list. Deliberately left under-scoped in the roadmap sketch until this beta got its own real planning conversation, across several rounds:

- Confirmed a **nook is the primary way a shelf is organized and shown to visitors** — not an optional side list layered on top. What a visitor sees on someone's profile *is* their nooks (plus an honest "unsorted" catch-all for anything not yet placed).
- Named after rejecting "playlist" — too suggestive of music and of sequence/reproduction, when the point is to be completely open to whatever the person wants it to mean. The author's own reference point: how personal and expressive an old Tumblr blog's theme used to feel, before feeds homogenized everything.
- Confirmed a nook needs its **own Lexicon record** (not a label field on `shelf.item`) — free ordering and a single shareable thing both require it.
- Confirmed a curated, non-free set of visual presentation choices (a handful of themes) rather than an open color picker — personalization without breaking the product's own visual coherence.

## Scope

1. **`social.orbita.shelf.nook`, a new Lexicon.** ✅ **Built.** [`lexicons/social/orbita/shelf/nook.json`](../lexicons/social/orbita/shelf/nook.json) — `name`, optional `description`, an ordered `works` array (same `{provider, id}` shape as `shelf.item#work`, reused rather than duplicated), and an optional `style.theme` from a small known set (`default`/`warm`/`cool`/`midnight`). Order is just array order — entirely the author's choice, nothing derived.
2. **Only works already on the shelf can go in a nook.** ✅ **Built.** Enforced by the appview at write time (`shelfItemExists` in [`nooks.go`](../cmd/appview/nooks.go)) — the Lexicon itself has no way to validate against another collection's contents.
3. **Editing is a whole-record replacement, not incremental.** ✅ **Built.** A nook has no separate membership record type — renaming, reordering, or adding/removing a work all just resend the full `works` array via `com.atproto.repo.putRecord`. This is the first record in this project that gets edited after creation, which meant the webhook had to start handling `action: "update"` events, not just `"create"` — scoped specifically to nooks for now, since nothing else has an edit path yet.
4. **The profile groups by nook.** ✅ **Built.** `GET /api/profile/{handle}` returns `nooks` (each with its own works resolved to title/poster) and `unsorted` (shelf items not in any nook) instead of one flat `shelf` array. Unsorted is never hidden or folded into a default grouping — a freshly shelved work sits there honestly until the person places it somewhere.
5. **Curated presentation themes.** ✅ **Built.** Four themes (`default`/`warm`/`cool`/`midnight`), implemented as CSS classes derived from the existing Observatório tokens (`color-mix` against `--signal`/`--surface`, no new hex values introduced) — personalization stays inside the product's own palette rather than opening a free color picker.
6. **Nook management lives on `/shelf`, presentation lives on `/profile`.** ✅ **Built.** Matches the split `/shelf`/`/profile` already established in Beta 4/5 — manage your own things privately in one place, see how they present publicly in another.
7. **Direct manipulation, reworked after first review (2026-07-18).** ✅ **Built.** The first pass used a form (name/description/theme fields, a checkbox list to pick works) — reviewed and replaced: creating a nook is now a single click plus a name, no other fields up front; populating and reordering one is native HTML5 drag-and-drop (no library) — drag a poster from Unsorted into a nook, drag within a nook to reposition, drag back out to un-sort it. Posters render everywhere works are picked or arranged, never plain text names. Nooks render above the page, the Unsorted grid below — the visual order matches "this is organized, this isn't" directly. A small "remove from shelf" control and a lightweight settings toggle (description/theme) stay available, just secondary to the drag interaction itself.

## Decisions made in planning conversation

- **"Nook," not "playlist."** Named after the author explicitly rejected "playlist" for presuming music/sequence, and wanted something as open as an old Tumblr theme. English chosen deliberately for this one (unlike most of this product's still-Portuguese vocabulary) — a call made mid-conversation, not a default.
- **A dedicated Lexicon record, decided before writing it** — a label field on `shelf.item` was considered and rejected: it can't give free ordering or a single shareable object, both explicitly wanted.
- **`nook_items` is fully rebuilt on every (re)index**, matching `putRecord`'s "whole record replaced" semantics, rather than tracked as incremental diffs.
- **Webhook now handles `"update"` actions — scoped to this need, not applied broadly.** Every other collection here still only ever arrives as `"create"` in practice (no edit path exists for them), so this isn't a blanket policy change, just closing the specific gap nooks introduced.
- **Same class of bug as Beta 1's "Titanic" incident, caught again.** The OAuth scope list in `oauth.go` never gained entries for `social.orbita.repost` or `social.orbita.shelf.nook` when those collections were added — nook creation failed against the real PDS with the same `ScopeMissingError` shape as the original delete-scope bug. Fixed by adding both, with nook explicitly requesting `create`, `update`, *and* `delete` — the first collection in this project needing all three from day one. Existing sessions can't be retroactively upgraded; signing out and back in is required to pick up the new scope, same as the original incident.
- **A second, separate incident: Tap's `TAP_COLLECTION_FILTERS` was also never updated.** After fixing the OAuth scope above, creating and editing a nook still didn't persist across a page reload — writes reached the real PDS fine, but neither running Tap instance (local sandbox or real relay) had `social.orbita.repost` or `social.orbita.shelf.nook` in its filter list, so the webhook never received those events at all; nothing to do with the "create/update" handling added to `webhook.go`, which was already correct. Both instances were restarted with the full four-collection list. Restarting Tap does **not** retroactively backfill records written while a collection was filtered out — the "Favs" nook created during the OAuth-scope bugfix testing had to be recreated by hand afterward. Lesson for next time a collection is added: updating `webhook.go` to handle it is necessary but not sufficient — every running Tap instance's `TAP_COLLECTION_FILTERS` needs the new NSID too, or the write silently never reaches the local index.

## Post-beta refinements (2026-07-18)

Beta 7 shipped as scoped above, then the author asked for three concrete improvements to the nook feature after using it for real — the shelf being "the heart of the product" was worth this extra pass.

8. **Nooks can be reordered by dragging.** ✅ **Built.** [`lexicons/social/orbita/shelf/nook.json`](../lexicons/social/orbita/shelf/nook.json) gained an optional `order` integer. Stored as `sort_order` in SQLite (`order` is a reserved SQL word) with **gapped positions** (multiples of 1000, via `nextNookOrder` in [`nooks.go`](../cmd/appview/nooks.go)) — moving one nook to a new spot only ever rewrites that one record, computing the midpoint between its new neighbors, rather than renumbering every sibling. This is the reason nook ordering needed its own field at all: the works *inside* one nook are just array order in a single record, but nooks themselves are separate records, so their relative order has nowhere else to live. Falls back to `created_at` for nooks written before the field existed (a real `NULL`, not a misleading `0`). Dragging a nook's title on `/shelf` (`frontend/shelf.js`) reorders it live.
9. **A single nook is shareable on its own**, not just visible embedded in a whole profile. ✅ **Built.** New `GET /profile/{handle}/nook/{rkey}` — the first **server-templated** page in this project ([`nookpage.go`](../cmd/appview/nookpage.go), `frontend/nook.html`), a deliberate deviation from the "static shell + client-side JSON fetch" pattern every other page uses. The reason is real Open Graph tags (`og:title`/`og:description`/`og:image`, the latter the first work's poster): a link unfurler in Slack, iMessage, or Bluesky itself never runs JavaScript, so those tags have to already be in the raw HTML response, not filled in after load by `nook.js`. The JSON the page's own script re-fetches (`GET /api/profile/{handle}/nook/{rkey}`) is the same data, for the person who actually opens the link. A `share ⤴` control (now a shared `shareButton()` helper in `common.js`, also used by note sharing) appears both on this standalone page and next to each nook on the normal profile page.
10. **Nook themes redesigned around duotone, not a soft color wash.** ✅ **Built.** Prompted by the author naming the shelf as the product's design centerpiece and pointing at art-history references rather than accepting the first pass: studied Bauhaus/Swiss grid discipline (already this product's own spine — the orbital mark, the type-role system), Blue Note's duotone album sleeves (Reid Miles unifying whatever mismatched source photo he was handed under one flat, two-color mood), and De Stijl's confident flat color blocking, explicitly rejecting soft gradients. The old `.nook-warm`/`-cool`/`-midnight` classes tinted the whole section background with `color-mix` — replaced with a bold flat top rule plus one small offset color block (never centered — the same placement rule the orbital mark itself already used for its affinity point), and each poster inside a themed nook is now duotoned via an SVG `feColorMatrix`/`feComponentTransfer` filter (`ensureDuotoneDefs()` in `common.js`, injected once per page load, no build step or canvas needed) — grayscale, then black remapped to the theme's shadow tone and white to its highlight tone. Hovering a poster lifts the duotone to show its real colors, so there's still something to find on closer look. Every poster on the shelf (themed or not) also picked up the same corner-crosshair instrument framing the work page's own hero poster already used, just scaled down — one visual vocabulary for "this is an observed object," not a card, wherever a poster appears. Applies identically on `/shelf` (organizing) and `/profile`/the standalone nook page (presentation) — what you arrange is exactly what gets shared.
11. **Three more themes, each from a distinct real print/design tradition rather than an invented color.** ✅ **Built.** `riso`, `indigo`, and `manifesto` added to `style.theme`'s `knownValues` alongside the original four. Researched rather than picked by eye: `riso` uses Teal and Fluorescent Pink, actual stocked Risograph ink colors (`#00838a`/`#ff48b0`, sourced from the community-maintained [`mattdesl/riso-colors`](https://github.com/mattdesl/riso-colors) reference) — Riso printing is duotone by physical construction, two flat inks overprinted, not an approximation of one; `indigo` is aizuri-e, the Edo-period ukiyo-e genre printed almost entirely in Prussian blue (popularized in Japan in the 1820s via the Dutch trading post at Nagasaki), historically reserved for contemplative landscape work rather than the jazz-club mood `warm`/`cool`/`midnight` already cover; `manifesto` is Soviet Constructivist poster design (Rodchenko/Lissitzky) — flag-red on near-black, the most urgent register in the set. Each is a full duotone pair (shadow + highlight) wired the same way as the original three: an SVG filter in `common.js`, a `--duo-*-hi` custom property in `styles.css`, and an entry in `frontend/shelf.js`'s `NOOK_THEMES`.

12. **A resync endpoint — because the local index kept silently falling behind the PDS.** ✅ **Built.** Root-caused twice in one afternoon of local development: restarting the appview to pick up new code leaves a short window where Tap's webhook `POST` comes back `connection refused`; Tap retries on its own schedule, and if the appview comes back up in time the record shows up late (confirmed from Tap's own log — `"webhook failed, retrying"`, `retries: 0` through `29`, over about 4.5 minutes, for one note that did eventually land), but if that window outlasts Tap's retry budget, the record is silently and permanently missing from the local index — confirmed by comparing `com.atproto.repo.listRecords` against `orbita.db` by hand and finding 8 nooks that existed for real on the PDS but had never been indexed. This is not a one-off bug, it's the actual shape of eventual consistency in this architecture: the local SQLite database is a cache of the PDS, and nothing before this made that promise hold under real timing conditions, only in the easy case. `POST /api/resync` ([`resync.go`](../cmd/appview/resync.go)) re-reads every record the signed-in account has, across all four indexed collections, straight from their own PDS (`com.atproto.repo.listRecords`, paginated), reconciles the local index against it — upserting anything missing, deleting anything indexed locally that's been removed at the source — and a "resync from PDS" button on `/shelf` triggers it by hand. `indexRecord`, the actual decode-and-write logic, was extracted out of the webhook handler into a shared function (`webhook.go`) so the live path and the resync path can never quietly diverge into two different ideas of how a record becomes a row. Scoped to the signed-in account's own data only — resyncing isn't something one account can trigger for another. Verified against a copy of the real database before running for real: it both backfilled records the webhook had missed and correctly pruned nooks that had genuinely been deleted at the source in the meantime.
13. **Nook size cap dropped from 200 to 50 works, with an actual justification this time.** ✅ **Built.** The author asked to revisit the original 200 figure and required it be well justified with "a number that makes sense," not picked by eye — so both sides got checked. Technical side: the AT Protocol spec's own recommended `MAX_CBOR_RECORD_SIZE` is 1 MiB; even at a deliberately pessimistic per-work size (a MusicBrainz UUID as the `id`, ~72 bytes per `{provider, id}` entry), a single nook record could hold roughly 14,500 works before hitting that ceiling — so 200 was never a technical constraint, it was a guess wearing technical clothing. The real constraint has to be the nook's own premise: a "small manifesto of taste," not the shelf again (that's what "Unsorted" already is, honestly, with no cap). Researched real curatorial conventions in the same cultural territory this product touches — the Sight & Sound poll's decades-old 10-title ballot, "best of the year/decade" lists clustering well under 50 — and landed on **50**: generous enough to not become friction, small enough that hitting it still reads as a real editorial choice. [`lexicons/social/orbita/shelf/nook.json`](../lexicons/social/orbita/shelf/nook.json)'s `works.maxLength` now says so explicitly, with the reasoning in its own description field, not just this doc. Enforced client-side too (`NOOK_WORKS_LIMIT` in `frontend/shelf.js`) since this appview never validates incoming records against the Lexicon on write — without that check, dragging past the cap would silently produce a record the schema itself calls invalid. Caught and fixed a real bug while adding the check: the original draft removed a dragged work from its source nook *before* checking whether the target nook was full, which would have made the work vanish entirely on a rejected move; the limit check now runs first.
14. **Rebuilt drag-and-drop so it actually reads as direct manipulation.** ✅ **Built.** The author's own words: dragging didn't visibly move anything, cards didn't shift out of the way before release, nothing like "Windows folders or a Spotify playlist." The underlying cause is a real limitation of native HTML5 drag-and-drop, not a bug: the browser's own drag image is a flat, semi-transparent snapshot with no styling control, and dragover never live-updates the DOM on its own — everything before this only computed a target position once, at drop. Fixed with two additions layered onto the same native API (still no library, matching this frontend's own no-build-step rule): (1) a custom drag ghost (`attachDragGhost`/`positionDragGhost`/`detachDragGhost` in `shelf.js`) — the native drag image is suppressed (`setDragImage` pointed at a blank canvas) and replaced with a real cloned element that tracks the cursor 1:1 via `position: fixed` + `transform: translate(...)`, updated on every `dragover` (not `drag`, which has an unreliable-coordinates quirk in Firefox); (2) a live reflow — the dragged card's actual DOM node is moved into its new position on every `dragover` that crosses a threshold, and FLIP (First-Last-Invert-Play: snapshot every sibling's position before the move, then animate each one from its old position to its new one) makes whichever cards had to shift visibly slide there instead of snapping. `dataTransfer.getData()` can't be read during `dragover` by design (a cross-origin-drag security restriction in the spec, unrelated to same-page drags but enforced regardless) — worked around with a module-level `dragState` object set at `dragstart` and read everywhere else, which works because the source and every target are the same page. A themed "arrived" animation (`.just-landed`, a keyframe ring using the destination nook's own `--frame-accent` custom property — the same one already driving that nook's duotone poster frames, so it's free, not a new color) plays once on a card's very next render when it lands somewhere new, tracked by a one-shot `justLanded` marker rather than a timer. The nook-works cap (item 13) is now enforced live during the drag itself — a full nook simply refuses to accept the card as a drop target, rather than accepting it and then rejecting on drop.
15. **Closed two real duplication bugs, and added a persistence-layer guarantee against a third.** ✅ **Built.**
    - **Duplicate nooks**: `renderNewNookTrigger` in `shelf.js` called `create()` from both a `keydown` (Enter) and a `blur` listener on the same input. Pressing Enter kicks off the (async) `POST /api/nooks`; when it resolves, `rerender()` wipes the whole tree, detaching that very input while it's still focused — which fires a native `blur` on it, re-entering `create()` a second time before anyone had a chance to notice, submitting the same name twice. Fixed by clearing `input.value` synchronously, before the first `await`, so the re-entrant call reads an empty string and takes the early-return path instead of submitting again.
    - **Duplicate shelf items**: `POST /api/shelf/add` had no idempotency check at all — every call minted a fresh `createRecord`, so re-adding a work already on the shelf (a second visit, a `/search` result that doesn't yet know what's already there) silently produced two separate `social.orbita.shelf.item` records for the same work. Fixed with a new `getShelfItem` lookup (`db.go`) checked before writing — an existing item is returned as-is instead of creating a second one.
    - **Duplicate works within a nook**: nothing stopped the same `{provider, id}` from appearing twice in one nook's `works` array — the Lexicon's own array type has no uniqueness constraint, and AT Protocol validates nothing server-side. Guarded at three layers rather than just one, since any of them could plausibly be the one that's ever wrong: client-side in `saveNook` (dedup before sending), server-side in `buildNookRecord` (dedup before writing the actual PDS record, `api.go`), and again in `insertNook` (`nooks.go`, dedup before indexing locally) — the last one is the real guarantee, since it applies to anything landing in the local index regardless of source (the live webhook, a resync read straight from the PDS), not just this appview's own write path.
16. **Shelf posters are clickable.** ✅ **Built.** `renderPoster` (`shelf.js`) was a plain `<div>` — no way to get from a card on `/shelf` to that work's own page without going through search again. Now an `<a href="/works/{provider}/{id}">`, draggable exactly as before (a native drag gesture and a plain click are already distinguished by the browser without any extra code) — but the nested "remove from shelf" button needed `e.preventDefault()` added alongside its existing `e.stopPropagation()`, since a click anywhere inside an `<a>` triggers navigation by default regardless of what a descendant's own click handler does.
17. **Profile page shows nook cards, not the whole shelf.** ✅ **Built.** With nooks capped at 50 works each (item 13), the profile page — which used to render every nook's *complete* works grid inline, one full poster grid per nook, plus a full "Unsorted" grid on top — could put well over a hundred posters on one page. Flagged directly: if the profile always shows everything, there's no real reason for the standalone shareable nook page (item 9, `nookpage.go`) to exist at all. Each nook is now a compact card (`renderNookCard`, `profile.js`) — up to 6 preview posters in a small mosaic, the name, a count — and the entire card is the link to that nook's own complete page; "see it all" is one click away, on the nook that earned it. The preview posters reuse the exact same `.shelf-grid-item` class the full grid uses, so they inherit the theme's duotone filter and `--frame-accent` for free — no new CSS rule needed for that part, just a smaller grid and the crosshair corners turned off (`.nook-mini-item::before/::after { content: none; }`, too fussy at that scale). "Unsorted" dropped from a full poster grid to a single muted text line (a count, no images) — an honest mention rather than another wall of posters undermining the whole point of a curated nook.

## Explicitly not in this beta

- Constellation/archetype awareness of nooks (Beta 8, next) — how those visualize a shelf organized this way is an open question for that beta, not resolved here.
- Any change to how `/shelf` (the private management list) itself looks — nooks organize the *public presentation*, not the private list.
