# Beta 7 ‚Äî planning draft

**Status: Beta 7 done ‚úÖ** (2026-07-18). Nooks exist ‚Äî a person's own shelf is now organized and presented through them, not a single flat grid.

## Goal

Flagged early (2026-07-17) as potentially **the apex of the whole product**: the shelf is the one surface entirely about a person's own taste and judgment, and it didn't reflect that at all ‚Äî just a flat, chronological list. Deliberately left under-scoped in the roadmap sketch until this beta got its own real planning conversation, across several rounds:

- Confirmed a **nook is the primary way a shelf is organized and shown to visitors** ‚Äî not an optional side list layered on top. What a visitor sees on someone's profile *is* their nooks (plus an honest "unsorted" catch-all for anything not yet placed).
- Named after rejecting "playlist" ‚Äî too suggestive of music and of sequence/reproduction, when the point is to be completely open to whatever the person wants it to mean. The author's own reference point: how personal and expressive an old Tumblr blog's theme used to feel, before feeds homogenized everything.
- Confirmed a nook needs its **own Lexicon record** (not a label field on `shelf.item`) ‚Äî free ordering and a single shareable thing both require it.
- Confirmed a curated, non-free set of visual presentation choices (a handful of themes) rather than an open color picker ‚Äî personalization without breaking the product's own visual coherence.

## Scope

1. **`social.orbita.shelf.nook`, a new Lexicon.** ‚úÖ **Built.** [`lexicons/social/orbita/shelf/nook.json`](../lexicons/social/orbita/shelf/nook.json) ‚Äî `name`, optional `description`, an ordered `works` array (same `{provider, id}` shape as `shelf.item#work`, reused rather than duplicated), and an optional `style.theme` from a small known set (`default`/`warm`/`cool`/`midnight`). Order is just array order ‚Äî entirely the author's choice, nothing derived.
2. **Only works already on the shelf can go in a nook.** ‚úÖ **Built.** Enforced by the appview at write time (`shelfItemExists` in [`nooks.go`](../cmd/appview/nooks.go)) ‚Äî the Lexicon itself has no way to validate against another collection's contents.
3. **Editing is a whole-record replacement, not incremental.** ‚úÖ **Built.** A nook has no separate membership record type ‚Äî renaming, reordering, or adding/removing a work all just resend the full `works` array via `com.atproto.repo.putRecord`. This is the first record in this project that gets edited after creation, which meant the webhook had to start handling `action: "update"` events, not just `"create"` ‚Äî scoped specifically to nooks for now, since nothing else has an edit path yet.
4. **The profile groups by nook.** ‚úÖ **Built.** `GET /api/profile/{handle}` returns `nooks` (each with its own works resolved to title/poster) and `unsorted` (shelf items not in any nook) instead of one flat `shelf` array. Unsorted is never hidden or folded into a default grouping ‚Äî a freshly shelved work sits there honestly until the person places it somewhere.
5. **Curated presentation themes.** ‚úÖ **Built.** Four themes (`default`/`warm`/`cool`/`midnight`), implemented as CSS classes derived from the existing Observat√≥rio tokens (`color-mix` against `--signal`/`--surface`, no new hex values introduced) ‚Äî personalization stays inside the product's own palette rather than opening a free color picker.
6. **Nook management lives on `/shelf`, presentation lives on `/profile`.** ‚úÖ **Built.** Matches the split `/shelf`/`/profile` already established in Beta 4/5 ‚Äî manage your own things privately in one place, see how they present publicly in another.
7. **Direct manipulation, reworked after first review (2026-07-18).** ‚úÖ **Built.** The first pass used a form (name/description/theme fields, a checkbox list to pick works) ‚Äî reviewed and replaced: creating a nook is now a single click plus a name, no other fields up front; populating and reordering one is native HTML5 drag-and-drop (no library) ‚Äî drag a poster from Unsorted into a nook, drag within a nook to reposition, drag back out to un-sort it. Posters render everywhere works are picked or arranged, never plain text names. Nooks render above the page, the Unsorted grid below ‚Äî the visual order matches "this is organized, this isn't" directly. A small "remove from shelf" control and a lightweight settings toggle (description/theme) stay available, just secondary to the drag interaction itself.

## Decisions made in planning conversation

- **"Nook," not "playlist."** Named after the author explicitly rejected "playlist" for presuming music/sequence, and wanted something as open as an old Tumblr theme. English chosen deliberately for this one (unlike most of this product's still-Portuguese vocabulary) ‚Äî a call made mid-conversation, not a default.
- **A dedicated Lexicon record, decided before writing it** ‚Äî a label field on `shelf.item` was considered and rejected: it can't give free ordering or a single shareable object, both explicitly wanted.
- **`nook_items` is fully rebuilt on every (re)index**, matching `putRecord`'s "whole record replaced" semantics, rather than tracked as incremental diffs.
- **Webhook now handles `"update"` actions ‚Äî scoped to this need, not applied broadly.** Every other collection here still only ever arrives as `"create"` in practice (no edit path exists for them), so this isn't a blanket policy change, just closing the specific gap nooks introduced.
- **Same class of bug as Beta 1's "Titanic" incident, caught again.** The OAuth scope list in `oauth.go` never gained entries for `social.orbita.repost` or `social.orbita.shelf.nook` when those collections were added ‚Äî nook creation failed against the real PDS with the same `ScopeMissingError` shape as the original delete-scope bug. Fixed by adding both, with nook explicitly requesting `create`, `update`, *and* `delete` ‚Äî the first collection in this project needing all three from day one. Existing sessions can't be retroactively upgraded; signing out and back in is required to pick up the new scope, same as the original incident.
- **A second, separate incident: Tap's `TAP_COLLECTION_FILTERS` was also never updated.** After fixing the OAuth scope above, creating and editing a nook still didn't persist across a page reload ‚Äî writes reached the real PDS fine, but neither running Tap instance (local sandbox or real relay) had `social.orbita.repost` or `social.orbita.shelf.nook` in its filter list, so the webhook never received those events at all; nothing to do with the "create/update" handling added to `webhook.go`, which was already correct. Both instances were restarted with the full four-collection list. Restarting Tap does **not** retroactively backfill records written while a collection was filtered out ‚Äî the "Favs" nook created during the OAuth-scope bugfix testing had to be recreated by hand afterward. Lesson for next time a collection is added: updating `webhook.go` to handle it is necessary but not sufficient ‚Äî every running Tap instance's `TAP_COLLECTION_FILTERS` needs the new NSID too, or the write silently never reaches the local index.

## Post-beta refinements (2026-07-18)

Beta 7 shipped as scoped above, then the author asked for three concrete improvements to the nook feature after using it for real ‚Äî the shelf being "the heart of the product" was worth this extra pass.

8. **Nooks can be reordered by dragging.** ‚úÖ **Built.** [`lexicons/social/orbita/shelf/nook.json`](../lexicons/social/orbita/shelf/nook.json) gained an optional `order` integer. Stored as `sort_order` in SQLite (`order` is a reserved SQL word) with **gapped positions** (multiples of 1000, via `nextNookOrder` in [`nooks.go`](../cmd/appview/nooks.go)) ‚Äî moving one nook to a new spot only ever rewrites that one record, computing the midpoint between its new neighbors, rather than renumbering every sibling. This is the reason nook ordering needed its own field at all: the works *inside* one nook are just array order in a single record, but nooks themselves are separate records, so their relative order has nowhere else to live. Falls back to `created_at` for nooks written before the field existed (a real `NULL`, not a misleading `0`). Dragging a nook's title on `/shelf` (`frontend/shelf.js`) reorders it live.
9. **A single nook is shareable on its own**, not just visible embedded in a whole profile. ‚úÖ **Built.** New `GET /profile/{handle}/nook/{rkey}` ‚Äî the first **server-templated** page in this project ([`nookpage.go`](../cmd/appview/nookpage.go), `frontend/nook.html`), a deliberate deviation from the "static shell + client-side JSON fetch" pattern every other page uses. The reason is real Open Graph tags (`og:title`/`og:description`/`og:image`, the latter the first work's poster): a link unfurler in Slack, iMessage, or Bluesky itself never runs JavaScript, so those tags have to already be in the raw HTML response, not filled in after load by `nook.js`. The JSON the page's own script re-fetches (`GET /api/profile/{handle}/nook/{rkey}`) is the same data, for the person who actually opens the link. A `share ‚§¥` control (now a shared `shareButton()` helper in `common.js`, also used by note sharing) appears both on this standalone page and next to each nook on the normal profile page.

## Explicitly not in this beta

- Constellation/archetype awareness of nooks (Beta 8, next) ‚Äî how those visualize a shelf organized this way is an open question for that beta, not resolved here.
- Any change to how `/shelf` (the private management list) itself looks ‚Äî nooks organize the *public presentation*, not the private list.
