# Beta 3 — planning draft

**Status: Beta 3 done ✅** (2026-07-17). All four scope items built and confirmed, plus real extra ground covered along the way: handle/avatar resolution, a shelf add/remove control on the work page itself, shareable per-note links, and dedicated episode pages that weren't part of the original sketch.

## Goal

Everything through Beta 2 proves the protocol mechanics work — OAuth writes, Tap indexing, cross-account aggregation, season/episode browsing — but every page is a raw `fmt.Fprintf` HTML string. That was the right call while the point was proving mechanism, but it's the wrong shape for anything presentation-driven going forward (profile especially, per the conversation that reordered this roadmap — you can't really discuss a profile page without being able to look at one). Beta 3 is where `orbita` gets a real interface, applied first to what already exists (shelf, work pages, season/episode navigation, notes), so every beta after this one inherits it instead of starting from raw strings again.

Before drafting scope, we researched how other open-source AT Protocol apps (Bluesky's own `social-app`, Frontpage, Smoke Signal, WhiteWind, Leaflet) approach this. Two findings shaped the plan below:

- **There's no shared visual convention in the Atmosphere ecosystem to align with.** Every app looks like its own independent product; the only thing they share is that the protocol itself (DIDs, PDS hosts) stays invisible to end users. So keeping Órbita's own distinct "Observatório" identity isn't going against ecosystem norms — every app already does this.
- **One thing to explicitly avoid**: Bluesky's web layout has a right-hand column for trending topics/suggested feeds — directly against this project's own principle 2 (no trending, no algorithmic surfaces). Worth naming as a conscious decision, not an oversight.
- What's worth borrowing isn't visual identity but **practices**: menu/nav placement across breakpoints, how a profile and a feed are laid out structurally, responsive behavior at different screen sizes, sensible CSS transition patterns — studied as engineering reference, not copied, same way `comum`'s own Observatório identity was always meant to be its own thing.

## Scope

1. **Split the appview into a JSON API.** ✅ **Done.** [`cmd/appview/api.go`](../cmd/appview/api.go) now carries the work page, season, episode, search, shelf add/delete, notes add, and `/api/me` as JSON endpoints. OAuth session handling and DPoP-bound tokens never left the server — the browser only ever calls same-origin JSON endpoints, authenticated by the existing cookie, exactly as planned.
2. **Writes become JSON endpoints too.** ✅ **Done.** `/api/notes/add`, `/api/shelf/add`, `/api/shelf/delete` all write via `fetch()`, no full-page reloads. Went a step further than planned: each write handler captures `com.atproto.repo.createRecord`'s own response and returns the created record directly, so the frontend can append it optimistically — the local SQLite index only catches up once Tap delivers the event, and re-fetching immediately after a write would race that lag.
3. **A new `frontend/`.** ✅ **Done.** Vanilla HTML/CSS/JS, no build step, self-hosted fonts, implementing the Observatório identity. `common.js` holds helpers shared across pages (`el`, `fetchJSON`, `currentViewer`) so each new page doesn't redefine them.
4. **Apply it to what already exists.** ✅ **Done**, and expanded on: the work page (poster → who has it on their shelf, with an add/remove control right there → seasons, expandable in place → notes about the work), search (`/search`, replacing the classic manual raw-id form entirely — see below), and notes both work-level and episode-level. One thing added that wasn't in the original sketch: **each episode got its own dedicated page** (big still, synopsis, that episode's own notes) instead of nesting further into the season's accordion — the author's call, since that's genuinely where a specific episode's discussion lives, not one more expand-in-place level. The classic `GET /shelf` debug listing (Beta 0's "prove the webhook indexed it" page) was deliberately left as raw HTML — it was never a real product surface, just a Beta 0 debug artifact.

## Decisions made in planning conversation

- **One process, not two.** `comum` splits its frontend (static files via nginx) from its API (api-gateway) as separate services; `orbita`'s appview serves the static frontend files itself — one Go binary, one process, no nginx. Simpler, and fits a single-developer hobby project better than mirroring `comum`'s topology out of habit. CORS never came up as a result — same-origin throughout.
- **The classic manual raw-id shelf form is retired, not converted.** `handleShelfAddForm`/`handleShelfAdd` (typing a provider and id by hand) were removed outright rather than reskinned — that path was the exact cause of the "Titanic" garbage-record incident in `BETA1-PLAN.md`, and `/search` plus `POST /api/shelf/add` fully replace what it did, with the TMDB-recognized-id guarantee search was built for in the first place. Caught a stale reference during testing: the post-login OAuth redirect still pointed at the now-dead `/shelf/add` — fixed to redirect to `/search` instead.
- **Handle/avatar resolution, added mid-beta, not in the original sketch.** Every DID shown in the UI (shelf accounts, note authors) resolves to a real handle and avatar instead of a raw DID string. Handle comes from the account's own DID document (`indigo/atproto/identity`, cached in memory); avatar is read from the account's own `app.bsky.actor.profile` record on its own PDS and resolved to a blob URL — deliberately not through Bluesky's `app.bsky.actor.getProfile` API, so it works the same for any AT Protocol account, not just ones registered with Bluesky specifically. Cached in a new `identity_cache` table, fail-open (falls back to a truncated raw DID) for accounts that genuinely can't be resolved, like disposable local sandbox test accounts.
- **Notes are individually shareable.** Each note gets a `#note-{rkey}` anchor and a share affordance (native `navigator.share()` where available, clipboard-copy fallback otherwise) — opening a shared link scrolls to and highlights that specific note.
- **"How literally to borrow from Bluesky's `social-app`" resolved itself in practice**: ended up staying at the level of general practice (native `<details>` for expand-in-place instead of a custom accordion, a responsive two-column hero, avoiding their trending rail on purpose) rather than reading their token/component source in detail — the Observatório spec already fully specifies this project's own tokens, so there was nothing to reconcile.

## Explicitly not in this beta

- Any new product surface — forum (Beta 4), events (Beta 5), profile (Beta 6) all come after, once the interface this beta builds already exists for them to use.
- Constellation/archetype visualization — explicitly deferred to whenever profile lands (Beta 6) or later.
- Real fan-out, affinity, feed, DMs — unrelated to this beta, still further out on `docs/ROADMAP.md`.
- The shelf-as-creative-space idea (custom label-playlists) — captured as its own future beta on `docs/ROADMAP.md`, deliberately not scoped here.

See [`docs/ROADMAP.md`](ROADMAP.md) for where this sits in the overall arc, and `docs/BETA0-PLAN.md`'s note on `comum` being a private repo for why earlier product decisions are referenced generically here rather than by file path.
